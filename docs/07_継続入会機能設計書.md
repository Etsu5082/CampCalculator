# 継続入会機能設計書

## 1. 概要

新学期に、前年度の会員が継続して入会する際のフローを簡略化する機能。

---

## 2. 現状の課題

- 毎年、全会員が新規入会フォームで全項目を再入力する必要がある
- 前年度のデータが活用されていない
- 変更点のみ更新したい

---

## 3. 解決策

### 3.1 年度管理の追加

**membersテーブルに年度カラムを追加**:

```sql
ALTER TABLE members
ADD COLUMN academic_year INT NOT NULL DEFAULT 2025 COMMENT '所属年度',
DROP INDEX student_id, -- 既存のUNIQUE制約を削除
ADD UNIQUE KEY unique_student_id_year (student_id, academic_year); -- 年度ごとにユニーク
```

### 3.2 継続入会フォーム

**URL**: `/renew`（公開ページ、認証不要）

**フロー**:
```
1. 名前で検索
   ↓
2. 前年度のデータを表示
   ↓
3. 「この内容で変更ありませんか？」
   ↓
4. 変更なし → そのまま新年度に登録
   変更あり → 編集画面で修正
   ↓
5. 確認画面
   ↓
6. 新年度の名簿に登録
```

---

## 4. データベース設計

### 4.1 membersテーブルの拡張

| カラム | 変更内容 |
|--------|---------|
| academic_year | 追加（INT, NOT NULL, DEFAULT 2025） |
| student_id | UNIQUE制約 → (student_id, academic_year)の複合UNIQUE |

### 4.2 年度管理テーブル（新規）

```sql
CREATE TABLE academic_years (
    id INT AUTO_INCREMENT PRIMARY KEY,
    year INT NOT NULL UNIQUE COMMENT '年度（例: 2025）',
    start_date DATE NOT NULL COMMENT '年度開始日',
    end_date DATE NOT NULL COMMENT '年度終了日',
    is_current TINYINT(1) NOT NULL DEFAULT 0 COMMENT '現在年度フラグ',
    enrollment_open TINYINT(1) NOT NULL DEFAULT 0 COMMENT '入会受付中フラグ',
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 7. 移行計画

### 7.1 既存データの移行

```sql
-- 既存データに2025年度を設定
UPDATE members SET academic_year = 2025;
```

### 7.2 新年度開始時の処理

1. 管理者が「新年度開始」ボタンをクリック
2. 現在年度のフラグを更新
3. 継続入会フォームを公開
4. 新入生用フォームも公開（並行）

---

## 8. 運用フロー

### 毎年4月

1. **管理者**: 新年度（2026年度）を開始
2. **新入生**: `/enroll` で新規入会申請
3. **継続会員**: `/renew` で継続入会（前年度データから）
4. **管理者**: 入会申請を承認
5. **管理者**: 前年度名簿は保持（過去データとして閲覧可能）

---
